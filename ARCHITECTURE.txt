TBCaptial Architecture
======================

Status: Conda environment implemented; data and backtest code not implemented.

Constraints:

* No S3, cloud database, NAS, shared DuckDB file, or remote storage service.
* AKShare is implemented first. Tushare only supplements or cross-checks an
  explicitly selected dataset; a partition never silently mixes providers.
* Research and backtest jobs never call AKShare or Tushare.
* The same monolith runs ingestion and backtest commands.


2. Local Storage and Optional Snapshot Transfer
------------------------------------------------

                    +---------------------------+
                    | Local Ingestion Process   |
                    | one writer + file lock    |
                    +-------------+-------------+
                                  |
                        validate + atomic rename
                                  v
                    +---------------------------+
                    | Local Data Root           |
                    | immutable snapshot_id     |
                    +-------------+-------------+
                                  |
                  manifest + exact files + SHA-256
                                  v
                    +---------------------------+
                    | Local DuckDB Views        |
                    | Research / Backtest       |
                    +---------------------------+

Optional machine transfer is explicit and offline:

  source data root -> export snapshot bundle -> copy by user
                   -> import into target local disk -> verify every hash
                   -> rebuild target DuckDB catalog

The runtime never connects to an external storage server. DuckDB is disposable;
Parquet files and manifests are authoritative.


3. Data Lifecycle and Snapshot Publication
------------------------------------------

 +---------------------+    +-----------+    +-----------+    +--------------+
 | Provider raw reply  | -> | Raw Batch | -> | Normalize | -> | Quality Gate |
 | AKShare or Tushare  |    | immutable |    | + lineage |    +------+-------+
 +---------------------+    +-----------+    +-----------+           |
                                                     +---------------+---------+
                                                     | PASS                   | FAIL
                                                     v                        v
                                    +------------------------+       +----------------+
                                    | Partition Manifest     |       | QUARANTINED    |
                                    | relative files + hashes|       | not queryable  |
                                    +-----------+------------+       +----------------+
                                                |
                                      atomic local publish
                                                v
                                    +------------------------+
                                    | Snapshot Manifest      |
                                    | status=PUBLISHED       |
                                    +-----------+------------+
                                                |
                                                v
                                    +------------------------+
                                    | DuckDB View / Backtest |
                                    +------------------------+

An upstream correction creates a new file, partition manifest, and child
snapshot. It never overwrites data referenced by an earlier experiment.


4. Backtest Public Interface
----------------------------

 +-------------------+      +-------------------+
 | BacktestRequest   |      | StrategySpec      |
 | snapshot/config   |      | type + parameters |
 +---------+---------+      +---------+---------+
           +--------------------------+
                      |
                      v
             +------------------+
             | run_backtest()   |
             +--------+---------+
                      |
                      v
 +-----------------------------------------+
 | AkQuantBacktestEngine adapter           |
 | SnapshotFeedAdapter                     |
 | TBCaptialStrategyBridge                 |
 | AShareExecutionGuard                    |
 | ResultTranslator                        |
 +--------------------+--------------------+
                      |
              embedded same-process call
                      v
 +-----------------------------------------+
 | AKQuant v0.3.2 backend                  |
 | Rust event loop / orders / account      |
 +--------------------+--------------------+
                      |
             translated read-only context
                      v
 +--------------------+--------------------+
 | Strategy lifecycle                      |
 | initialize -> start -> before_trading    |
 | -> on_bars -> order/fill/reject -> after |
 | -> stop                                 |
 +--------------------+--------------------+
                      |
                      v
 +-----------------------------------------+
 | BacktestResult                          |
 | summary + immutable artifact accessors  |
 +-----------------------------------------+

Formal runs accept snapshot_id, not a full-future DataFrame. StrategyContext
provides as-of history, account views, and target-portfolio commands only.
Only the internal bridge imports or inherits AKQuant types. Version, commit,
wheel SHA-256, compatibility tests, and A-share rejection semantics are gates.


5. Backtest Event Order for Trading Day T
-----------------------------------------

 +------------------+     +------------------+     +------------------+
 | Apply actions /  | --> | Unlock sellable  | --> | before_trading   |
 | security status  |     | T+1 positions    |     | read-only        |
 +------------------+     +------------------+     +--------+---------+
                                                           |
                                                           v
 +------------------+     +------------------+     +------------------+
 | order/fill/reject| <-- | Ledger update    | <-- | Execute pending  |
 | callbacks        |     | cash + position  |     | orders at open   |
 +--------+---------+     +------------------+     +------------------+
          |
          v
 +------------------+     +------------------+     +------------------+
 | Mark at close    | --> | Frozen BarSlice  | --> | Strategy creates |
 | as-of T          |     | all symbols once |     | target intent    |
 +------------------+     +------------------+     +--------+---------+
                                                           |
                                                           v
 +------------------+     +------------------+     +------------------+
 | after_trading    | <-- | Persist + check  | <-- | Plan orders for  |
 | read-only        |     | invariants       |     | T+1 open         |
 +------------------+     +------------------+     +------------------+

Data through the close of T may create an intent for the open of T+1. A single
BarSlice contains the complete expected universe for T, so cross-sectional
strategies never depend on per-symbol callback arrival order.
