# TBCaptial 设计与开发计划

文档状态：Draft v0.1

最后更新：2026-07-17
当前仓库状态：环境、AKShare Raw 小切片和 AKQuant `v0.3.2` 最小 adapter 已验收；Silver、正式 snapshot 和完整回测链路待实现

## 1. 执行摘要

TBCaptial 的第一目标不是覆盖所有量化场景，而是在双人团队内建立一个**可信、可复现、可快速迭代的 A 股日频回测闭环**。

整体选择 Python 模块化单体：一个仓库、一个 Python 包、一个命令行入口、一个默认进程。内部按数据、研究、策略、组合、回测、报告和基础设施分模块，模块间通过领域对象和接口协作。首版不引入微服务、消息队列或分布式调度。

推荐按以下顺序投入：

1. 固定时间语义、数据模型和回测契约；
2. 打通小股票池的数据同步与快照；
3. 固定并封装 AKQuant backend，补齐可人工验算的 A 股执行与账本语义；
4. 接入一个简单策略，输出完整实验产物；
5. 扩展到全市场、多因子和批量实验；
6. 最后才做模拟盘、实盘和性能专项优化。

## 2. 已确定的架构决策

| 决策 | 当前选择 | 原因 |
| --- | --- | --- |
| 系统形态 | 模块化单体 | 双人团队维护成本最低，调用链和回测状态容易调试 |
| 主要语言 | Python 3.11 | AKShare 官方推荐版本，研究生态完整；热点以后再针对性优化 |
| 环境管理 | Conda + `environment.yml` + 实际包清单 | 使用单一声明跨机器创建；当前只验收实际开发机，不生成多平台 lock |
| 首期频率 | 日频 | 最快验证策略闭环，同时降低撮合和数据复杂度 |
| 首期资产 | A 股现货、只做多 | 聚焦最主要使用场景，不提前引入保证金和衍生品语义 |
| 数据源 | AKShare 主、Tushare 辅，经 provider adapter 隔离 | 优先完成无 token 主链路，辅助源只补字段或交叉校验 |
| 权威存储 | 单机本地文件系统 | 零外部服务依赖，Parquet 与 manifest 保持可追溯和可复制 |
| 数据文件 | Parquet | 列式压缩、跨工具兼容、适合按日期分区 |
| 本地查询 | 本机 DuckDB catalog 直接查询本地 Parquet | 单机零服务运维，catalog 可从 manifest 重建 |
| 回测 backend | AKQuant `v0.3.2` 经 TBCaptial adapter 封装 | 复用 Rust/Python 事件循环与账户能力；策略不依赖其类型 |
| 回测模式 | 日频事件驱动，研究计算可向量化 | TBCaptial 固定 A 股接口和语义，AKQuant 是可替换的进程内实现 |
| 策略输出 | 目标权重/目标持仓，不直接操纵账户 | 分离选股、组合构建和执行，可统一做约束与归因 |
| 默认信号/成交 | T 日收盘后决策，T+1 日开盘成交 | 时间边界清晰，首版容易避免未来函数 |
| 数据版本 | 不可变分区 + 快照清单 | 保证历史修订后旧实验仍可解释和重放 |
| 性能策略 | 先 profiling，后 Polars/Numba/C++ | 避免凭经验提前制造双语言和构建复杂度 |

以上决策发生变化时，需要记录：问题、选择、替代方案、结果和迁移成本。影响回测结果的决策不得只在聊天或代码评论中出现。

## 4. 总体架构

### 4.1 数据与控制流

系统的主路径为：

`数据源 → 采集 → 原始层 → 标准化/校验 → 数据快照 → 特征/股票池 → 策略 → 组合构建 → 订单 → 撮合 → 账户 → 评估 → 实验产物`

所有步骤由同一个应用层编排。长任务可以在本机独立进程中运行，但仍属于同一代码库和同一发布单元，不因此拆服务。

### 4.2 内部依赖方向

- **领域层**定义证券、交易日、行情、目标组合、订单、成交、持仓、现金和账本规则；不依赖 AKShare、Tushare、AKQuant、DuckDB、CLI 或报告库。
- **用例层**编排“同步数据”“构建快照”“运行回测”“生成报告”等操作；依赖领域接口。
- **研究与回测层**实现特征、策略、组合契约、AKQuant backend 端口和绩效计算；只通过数据接口读取指定时间截面的数据。
- **基础设施层**实现 AKShare/Tushare、AKQuant、Parquet、DuckDB、文件系统、日志和报告适配器。
- **入口层**只负责解析配置、初始化依赖和调用用例，不包含业务规则。

禁止的依赖包括：策略直接调用 AKShare/Tushare、策略继承或返回 AKQuant 类型、撮合直接查询全量未来 DataFrame、指标模块修改账户、模块通过可变全局单例交换状态。

### 4.3 计划模块

| 模块 | 职责 | 不负责 |
| --- | --- | --- |

## 5. 核心契约

### 5.1 时间契约

每个数据对象至少区分以下时间：

- `event_time`：市场事件实际发生时间，例如交易日收盘；
- `publish_time`：数据源或公司对外发布时间；
- `available_time`：本系统在策略中允许使用该数据的最早时间；
- `ingested_at`：本系统实际采集时间；
- `as_of`：某次查询或决策允许看到的时间上限。

日线在 T 日收盘后才完整。默认策略在 T 日收盘事件完成后运行，订单最早在 T+1 开盘尝试成交。财务数据以后必须按公告日/实际可得日进入股票视图，不能按报告期直接前填。

任何接口如果无法说明 `as_of`，就不能进入策略路径。

### 5.2 策略契约

策略输入是只读上下文，包含：

- 决策时点和可用交易日历；
- 当时有效的股票池及每只股票的排除原因；
- 截止该时点可见的行情与特征；
- 当前总持仓、可卖持仓、现金和风险状态；
- 策略自身显式状态。

策略输出优先采用目标权重或目标持仓，并附带原因、置信度和有效期。策略不直接扣现金、不伪造成交、不绕过风险约束。

### 5.3 组合与订单契约

组合构建负责把策略意图转换为受约束目标：总仓位、单票上限、行业上限、现金缓冲和换手限制。订单规划再根据现有持仓、可卖数量、整手和预估费用生成订单。

订单必须有稳定标识、策略/实验来源、创建时间、最早执行时间、方向、数量、价格类型和有效期。所有拒绝、缩量、过期与取消都有机器可读原因。

### 5.4 数据源契约

AKShare 和 Tushare 只实现 provider 端口，AKShare 是默认主源。供应商原始字段按 source 分开落地，再由标准化步骤统一：

- 证券代码格式；
- 日期与时区；
- 成交量从接口声明单位转换为“股”；
- 成交额从接口声明单位转换为“元”；
- 空值、枚举和数值精度；
- 来源、采集时间和原始分区标识。

标准化后不保留供应商特有命名作为领域接口的必需部分。provider 和端点血缘必须保留；同一分区不得静默混源。

## 6. 数据架构

### 6.1 存储结论与选型理由

最终选择是：

**本机文件系统保存权威数据，Parquet 保存表型事实数据，JSON/Parquet 保存 manifest，DuckDB 直接查询本地文件。第一阶段不使用云存储、NAS 或数据库服务器。**

量化系统仍然是一个模块化单体；存储通过内部 `StorageBackend` 端口访问，但第一阶段只实现本地文件系统后端。保留端口是为了隔离领域逻辑与文件布局，不代表计划接入云服务。

| 方案 | 优点 | 问题 | 结论 |
| --- | --- | --- | --- |
| CSV + 目录 | 最简单、可人工查看 | 类型弱、体积大、查询慢、schema 易漂移 | 只允许小型测试夹具 |
| DuckDB 单文件保存全部事实 | SQL 方便、单机性能好 | 大文件备份和不可变快照边界不自然 | 只用作本地可重建 catalog |
| PostgreSQL | 事务、多写者和在线查询成熟 | 维护服务，列式研究扫描和大批量快照不如文件湖直接 | 实盘状态或多写控制面出现后再评估 |
| ClickHouse | 大规模列式查询和并发强 | 部署运维与数据复制成本超出双人日频 P0 | 当前不采用 |
| 本地 Parquet + manifest + DuckDB | 零服务运维、可版本、可直接分析、可离线复制 | 需要维护 manifest、原子发布和小文件治理 | **采用** |

不选择“每天临时调用 AKShare/Tushare 后直接回测”。它们是外部采集源，不是研究期在线数据库；策略和回测不得持有 provider client。

### 6.2 物理拓扑与权威边界

数据路径为：

`AKShare（主）/Tushare（辅）→ 单一本地采集进程 → staging → 质量检查 → 已发布快照 → DuckDB 视图 → 因子/回测`

各组件的权威性：

- **本地数据根目录**是当前机器市场数据、快照清单和正式实验产物的唯一权威来源；
- **Raw 文件和 snapshot manifest** 是恢复的核心，原则上不可修改；
- **本地 DuckDB catalog** 只保存可重建视图、索引和临时查询结果，不是权威事实库；
- **采集状态库**只帮助断点续跑，最终是否可用以本地发布 manifest 为准；
- **AKShare/Tushare**只在采集期间访问，本地快照就绪后回测不需要连接任何 provider。

需要换机器时，使用导出命令把 snapshot manifest 及其引用文件打包，再通过用户选择的离线方式复制到目标机；导入必须验证相对路径、字节数和 SHA-256。系统不自动连接外部云服务器。

### 6.3 三层数据和产物层

1. **Raw/Bronze**：按 provider 分开保存原始返回字段，只追加不覆盖；同时保存请求元数据、响应 hash 和采集时间，用于审计、修订对比和重放。
2. **Normalized/Silver**：统一 schema、单位、主键、时区、空值和质量状态；回测的市场事实来自这一层。
3. **Curated/Gold**：特征、历史股票池、横截面、基准和策略就绪数据；必须能从指定 Silver 快照与计算代码重建。
4. **Artifacts**：正式回测的 manifest、配置、净值、订单、成交、持仓、报告和日志；与市场数据分开设置生命周期。

Silver 不依赖 Gold，策略不读取 Raw。Raw 到 Silver、Silver 到 Gold 都记录输入快照、转换版本和输出 hash。

### 6.4 计划中的文件布局与分区

逻辑布局如下，实际本地数据根目录通过配置提供：

```text
tbcaptial-data/
├── raw/
│   └── source=<akshare|tushare>/endpoint=<api>/ingest_date=<date>/batch=<id>/
├── silver/
│   └── dataset=<name>/schema_version=<n>/trade_year=<yyyy>/trade_month=<mm>/
├── gold/
│   └── dataset=<name>/definition_version=<hash>/trade_year=<yyyy>/trade_month=<mm>/
├── manifests/
│   ├── partitions/
│   ├── datasets/
│   └── snapshots/<snapshot_id>/manifest.json
├── quality/<run_id>/
├── artifacts/<run_id>/
└── staging/<writer_id>/<transaction_id>/
```

分区规则：

- Raw 按 provider/endpoint、采集日期和批次组织，市场日期作为列和必要的子分区；
- Silver/Gold 日频大表首先按交易年、交易月分区，文件内部包含多只股票；
- 不按 symbol 建一级分区，否则数千股票会制造大量小目录和小文件；
- 当日增量可以先写日文件，定期合并为月度大小合适的文件；合并产生新文件和新 manifest，不原地修改仍被快照引用的文件；
- 小型主数据允许一个版本一个文件；全市场特征按定义版本和日期分区；
- Parquet 默认使用压缩，具体压缩算法、row group 和目标文件大小在性能测试后固定为数据集规范；
- schema 版本进入路径和 manifest。破坏兼容的变化创建新版本，禁止让读取方猜测列语义。

路径只负责粗粒度裁剪，manifest 才是快照文件列表的权威来源。回测不能用宽泛 glob 自动吸入后来新增的分区。

### 6.5 P0 数据集

| 数据集 | 逻辑主键 | 关键字段 | 用途 |
| --- | --- | --- | --- |
| `instrument` | `symbol, valid_from` | 市场、上市/退市日、状态 | 证券主数据和历史资格 |
| `trade_calendar` | `exchange, calendar_date` | 是否开市、前一交易日 | 驱动时钟 |
| `daily_bar` | `symbol, trade_date` | OHLC、昨收、量、额 | 信号、估值、撮合 |
| `adjustment_factor` | `symbol, trade_date` | 复权因子 | 构造稳定复权序列 |
| `daily_price_limit` | `symbol, trade_date` | 涨停价、跌停价 | 可成交判断 |
| `suspension` | `symbol, trade_date` | 停复牌类型、时间段 | 区分停牌和数据缺失 |
| `benchmark_bar` | `index_code, trade_date` | OHLC、收益 | 基准和超额收益 |
| `dataset_snapshot` | `snapshot_id` | 分区清单、hash、schema、构建时间 | 实验可复现 |

股票基础信息必须分别采集上市、暂停上市和退市状态，不能只保存当前上市股票，否则会产生幸存者偏差。

### 6.6 Provider 回填、增量与修订策略

只有采集用例访问 AKShare/Tushare，运行分为三种：

1. **首次历史回填**：先用 AKShare 下载交易日历和股票清单，再按 symbol/日期区间拉取未复权日线、基准、停复牌和公司行为；每个请求立即保存 Raw。随后才用 Tushare 辅助历史状态、复权因子和历史涨跌停。
2. **每日增量**：确认交易日和各接口数据已就绪后，拉取当日必需端点，校验完成才发布新快照。固定时间只是触发器，响应行数和跨表完整性才是就绪依据。
3. **修订回看**：每日重查最近若干交易日，周/月任务扩大回看窗口；内容 hash 变化时保存新 Raw 版本、生成差异报告和新 Silver 分区，不覆盖旧版本。

具体规则：

- AKShare 主数据低频全量刷新；历史上市、暂停和退市状态缺口由 Tushare 辅助分区补充；
- 交易日历按年度刷新，允许上游修订；
- AKShare 日线按 symbol/日期区间拉取；Tushare 辅助分区按其接口推荐粒度拉取；
- 基准行情按指数和日期范围拉取；
- 每个请求记录 provider、接口、非敏感参数、响应行数、耗时、重试、SDK 版本和响应 hash；
- 限流和重试参数配置化；权限不足、限流和网络失败分别分类；
- 空响应必须结合交易日历、接口权限和其他端点判断，不能直接解释为“没有数据”；
- 回填进度以成功落地的 Raw batch 和已发布 Silver 分区为准，可从任意未完成交易日恢复。

默认修订窗口按 provider 分别配置，不写死在领域代码中。

### 6.7 分区发布协议与快照

本地文件系统使用“同一文件系统 staging + 原子重命名 + manifest”的发布协议：

1. 单写者为一次采集创建 `transaction_id`，内容写入同一数据卷的 staging；
2. 计算文件大小、行数、schema、最小/最大键和 SHA-256；
3. 执行数据质量检查，失败文件进入隔离状态，不对研究可见；
4. 写分区 manifest，列出全部相对路径、hash、来源批次和质量结果；
5. 持有本机排他文件锁，将验证后的目录原子重命名到最终位置并排他创建发布标记；
6. 将一组已发布分区组装为 snapshot manifest；
7. 只有 snapshot 状态为 `PUBLISHED` 后，研究端才允许查询。

发布状态固定为 `STAGING → VALIDATED → PUBLISHED`，失败进入 `QUARANTINED`。任何已经进入快照的文件都不原地覆盖；上游修订生成新文件、新分区 manifest 和子快照。

一次 snapshot manifest 至少包含：

- snapshot id、父 snapshot id 和创建时间；
- 数据集、逻辑分区、相对文件路径、字节数、行数和内容 hash；
- schema 版本和转换代码版本；
- provider endpoint、采集批次和 Raw 血缘；
- 数据覆盖日期、质量结果、已知缺陷和人工例外；
- manifest 自身 hash 与发布者。

回测只接受 snapshot id，不接受“读取目录里的最新文件”。snapshot id 建议由时间前缀加 manifest 内容 hash 构成，既便于人类定位也能检测篡改。

### 6.8 数据质量门

每个 Silver 分区发布前至少检查：

- schema、类型、必填字段和枚举；
- 主键唯一性及相邻版本重复；
- OHLC 关系与非负价格/成交量；
- 交易日是否在日历中；
- 上市日期、退市日期和行情日期关系；
- 停牌与缺失行情的对应关系；
- 昨收、涨跌幅和收盘价的一致性容差；
- 涨跌停上下界和行情价格关系；
- 全市场当日行数相对近期分布的异常；
- 复权因子的缺失、倒退或异常跳变；
- 跨数据集 symbol 集合差异；
- Raw 行数到 Silver 行数的差异和每条过滤原因；
- 文件实际 hash 与 manifest hash 一致。

质量结果分为 `PASS`、`WARN`、`FAIL`。回测默认只消费 `PASS` 快照；`WARN` 需要显式允许并写入报告；`FAIL` 不得发布。人工白名单必须有原因、负责人、有效期和 issue，不能临时跳过后不留痕迹。

### 6.9 本地数据与查询目录

每台机器使用独立的本地数据目录：

```text
var/
├── data/                    # 权威本地数据根目录
├── catalog/catalog.duckdb   # 可重建的查询目录
├── work/                    # staging 与临时文件
└── runs/<run_id>/           # 实验产物
```

读取流程：读取本地 snapshot manifest，校验状态、相对路径和每个文件的 SHA-256；全部文件就绪后再建立 DuckDB 视图。导入外部复制的快照包时执行相同校验，半复制目录不会被标记为可用。

DuckDB catalog 由本地 manifest 重建，主要保存：

- 指向指定 Parquet 文件列表的视图；
- dataset/schema/snapshot 元数据；
- 本地质量结果和可选的临时物化结果；
- 查询统计与缓存命中信息。

研究 API 通过 dataset + snapshot + as-of 查询，不暴露物理路径。策略运行前完成快照解析；回测过程中不自动切换到更新快照。

本地清理使用保留期/配额，但正在运行、被正式实验引用或显式 pin 的快照不能删除。删除前必须检查 manifest 引用。

### 6.10 并发、写入所有权和故障恢复

团队采用“单写者、多读者”：

- 任一时间每个逻辑数据集只有一个采集发布者；
- 本机排他文件锁防止两个采集进程同时发布同一逻辑分区；
- 研究进程只读取 `PUBLISHED` manifest，不读取 staging；
- 多个研究进程可以读取同一本地已发布快照，各自使用只读连接或独立 DuckDB catalog；
- 不把 `catalog.duckdb` 放在 NAS/同步盘上由多个进程读写；DuckDB 官方并发模型以单进程写为主，共享网络目录需要额外谨慎；
- 采集进程崩溃后，过期 staging 可清理，已写 Raw 保留，依据 batch manifest 从断点恢复；
- 发布者崩溃若未写最终 manifest，则该批次对研究不可见；若 manifest 已发布，则内容必须完整且 hash 可验证。

P0 不引入外部队列或分布式锁服务。本地文件锁和单写者约束足够支持一个计划采集任务；出现真实多写需求后再评估事务型 catalog。

### 6.11 数据保留、备份与恢复演练

| 数据类别 | 默认保留 | 是否可重建 | 处理原则 |
| --- | --- | --- | --- |
| Raw provider 数据 | 长期保留，受数据授权约束 | 可能可重拉，但有修订、限流和下线风险 | 最高备份优先级，不覆盖 |
| Published Silver | 只要被任一快照引用就保留 | 可从 Raw 重建 | schema 迁移生成新版本 |
| Gold/特征缓存 | 活跃定义和正式实验引用长期保留 | 可从 Silver + 代码重建 | 可按配额清理未引用版本 |
| 正式实验产物 | 长期保留 | 部分可重跑 | manifest、指标、交易账本和报告不可丢 |
| 临时实验/查询 | 配置 TTL | 可重建 | 自动清理 |
| 本地 DuckDB catalog | 按磁盘配额 | 完全可重建 | 不进入权威备份 |

备份策略：

- 本地权威目录使用只追加策略和文件权限降低误删风险；
- Raw、snapshot manifest 和正式 artifacts 可定期增量复制到另一块本地磁盘，运行时不读取备份目录；
- manifest 和小型元数据可以更高频备份，Parquet 文件按增量复制；
- 定期做恢复演练：从空机器创建 Conda 环境，只使用本地备份副本和一个 snapshot id，恢复黄金回测；
- 记录恢复点目标和恢复时间目标，M1 先验证“可恢复”，M4 再按实际数据量设数值；
- 删除遵守各 provider 授权、团队保留政策和 manifest 引用检查，禁止按目录手工递归清理权威数据。

### 6.12 复权与公司行为

供应商前复权序列可能随查询区间或上游修订变化。为保证实验稳定：

- 永久保存未复权 OHLC 和独立复权因子；
- 因子/信号可使用系统按固定公式构造的复权序列；
- 撮合始终使用当日真实未复权价格与涨跌停价格；
- 账户估值以未复权价格和真实持仓为准；
- 分红、送转、配股等公司行为最终通过账户事件处理，不能只靠复权价格掩盖；
- 在公司行为模块完成前，P0 回测报告必须标注适用区间和近似方式，不能把近似结果用于生产判断。

### 6.13 存储升级触发条件

当前方案不是拒绝数据库，而是延后到需求出现。满足以下任一条件再评估 PostgreSQL、DuckLake、ClickHouse 或专用时序系统：

- 多个进程必须并发写同一逻辑表；
- 实盘订单状态要求低延迟事务与强一致更新；
- 大量用户并发做交互式查询，单机 DuckDB 无法满足；
- 分钟/Tick 数据量使 Parquet 小文件、增量更新或延迟不再可接受；
- 权限、审计或数据治理要求集中式 catalog。

升级时保持领域 `MarketDataRepository` 与 `StorageBackend` 接口不变，策略和回测不感知底层迁移。

## 7. 回测设计

回测底层固定使用经版本校验的 AKQuant `v0.3.2`，但公开接口和策略只依赖 TBCaptial。`AkQuantBacktestEngine` 负责把 snapshot feed、策略生命周期、目标组合、A 股执行守卫和结果 schema 映射到 AKQuant；只有内部 bridge 可以继承 AKQuant `Strategy`。AKQuant 版本/commit/wheel hash 是 run manifest 的一部分，升级必须通过全部黄金测试。

TBCaptial 仍拥有停牌/涨跌停的终态拒单、多日期公司行为、带生效日费用、完整 `BarSlice` 和不可变产物语义。AKQuant 扩展点不能满足时，适配器必须 fail-fast 或对固定 backend 做最小、可测试的上游扩展，禁止在用户策略中散落 workaround。第一阶段的精确接口见 `PLAN_LATEST.md` 第 6 章。

### 7.1 每日事件顺序

默认一个交易日按固定顺序执行：

1. 推进交易日时钟并结转上一日状态；
2. 处理当日生效的公司行为和证券状态变化；
3. 解锁符合交收规则的可卖持仓；
4. 在开盘事件尝试执行前一日生成的有效订单；
5. 更新成交、费用、现金和持仓；
6. 处理日频容量或剩余订单规则；
7. 在收盘价进行持仓估值；
8. 发布截止收盘可见的数据和特征；
9. 策略生成目标意图；
10. 组合构建、风控和订单规划；
11. 记录当日账本、净值、风险、拒单和审计事件；
12. 原子化保存检查点或完成本次运行。

事件顺序是公共契约。调整顺序必须更新黄金测试并说明回测结果迁移影响。

### 7.2 A 股首版交易规则

- 普通 A 股默认买入数量是 100 股整数倍；卖出零股按交易所规则一次性处理；
- 当日买入的普通 A 股在交收前不可卖出，账户同时维护总数量和可卖数量；
- 停牌订单不可成交，并区分“确认停牌”和“行情缺失”；
- 买入价触及且无法穿越涨停时拒绝或保持订单，卖出跌停同理；
- 新股前若干交易日、主板/科创板/创业板、ST 等不同价格限制必须由证券状态和生效日规则决定，不能只写一个固定百分比；
- 首版使用下一开盘价加可配置滑点，成交价必须受当日价格区间和涨跌停边界约束；
- 成交量参与率限制可配置，默认不允许策略成交量占当日成交量的异常比例；
- 现金、持仓和可卖数量不足时按明确策略拒绝或缩量，禁止负现金和裸卖空；
- 同一时点多个订单的现金竞争使用确定性排序，保证重复运行结果一致。

仅有日线时无法判断涨跌停板上的真实排队与盘中成交路径。系统应采用明确、保守、可配置的近似，并在报告中披露，不能宣称达到逐笔撮合精度。

### 7.3 费用模型

费用按市场、证券类型、方向、账户/券商和生效日期查表，包括：

- 券商佣金及单笔最低佣金；
- 卖出侧证券交易印花税；
- 过户费、经手费或其他需要纳入的费用；
- 滑点和市场冲击成本。

当前费率只作为配置初值，不作为永久常量。回测跨越规则变更日时自动选择当日有效规则。每笔成交保存费用拆分，日终账本可以逐项求和对账。

### 7.4 账户不变量

每个事件后至少保证：

- 现金、冻结现金、持仓总量和可卖量有明确关系；
- 只做多模式下持仓数量不为负；
- 可卖数量不大于总数量；
- 订单成交数量不超过订单剩余量；
- 成交、费用和现金变动可由复式式流水复算；
- 期末权益等于现金加各持仓市值及明确的其他应收应付；
- 每日权益变动可分解为市场盈亏、已实现盈亏、现金流和费用；
- 相同输入下事件序列和浮点舍入结果稳定。

资金账本建议以 Decimal 或最小货币单位记账；研究矩阵可以使用 float64。两者边界必须显式转换和统一舍入。

### 7.5 绩效与诊断

P0 报告至少输出：

- 累计/年化收益、年化波动、Sharpe、最大回撤、Calmar；
- 基准收益、超额收益和相对回撤；
- 总换手、平均持仓数、仓位、现金占比和费用占比；
- 月度/年度收益表；
- 订单数、成交数、拒单数、涨跌停/停牌/T+1 拒绝分布；
- 每日净值、持仓、订单、成交、费用和现金流水；
- 策略参数、数据快照、代码版本、运行环境和已知限制。

单个 Sharpe 不是验收标准。报告必须让开发者定位收益来自哪里、哪些订单没有成交以及结果是否受到数据缺陷影响。

## 8. 研究与策略迭代

### 8.1 推荐工作流

1. 在受控数据快照上探索想法；
2. 把有效逻辑移入版本化特征和策略模块；
3. 运行因子级检查：覆盖率、分布、稳定性、IC、分组收益和换手；
4. 通过策略接口生成目标组合；
5. 使用统一回测内核加入可交易性和费用；
6. 做样本内、验证集、样本外和不同市场阶段测试；
7. 保存完整实验产物并与基线比较；
8. 由另一人复核时间边界、数据选择和交易假设。

Notebook 可以用于探索，但不能成为生产策略的唯一实现或唯一实验记录。可复用逻辑、配置和测试必须进入包内模块。

### 8.2 首条基线策略

首条策略建议是简单且容易人工理解的横截面动量：

- 每周或每月调仓；
- 在一个小型固定股票池中按过去一段时间收益排序；
- 排除当时不可交易证券；
- 等权持有前 N 只；
- T 日收盘计算，T+1 开盘执行；
- 不做行业中性、优化器或机器学习。

这条策略的价值是覆盖多标的、排序、调仓、卖出、现金竞争、整手、T+1 和费用，而不是作为投资建议。均线择时可作为第二条单标的黄金策略。

### 8.3 实验产物

每次正式运行生成独立目录或对象，至少包含：

- `manifest`：run id、父实验、开始/结束时间和状态；
- 完整解析后的配置与配置 hash；
- Git commit、是否有未提交变更、Python 和依赖版本；
- 数据快照 id 与质量报告；
- 随机种子；
- 指标、净值、持仓、订单、成交、费用和拒单；
- 日志、异常和性能统计；
- 人类可读报告。

正式对比只比较状态为成功且数据质量门一致的实验。

## 9. 配置、运行与安全

### 9.1 配置层次

配置优先级建议为：默认值 < 项目配置 < 策略配置 < 命令行覆盖。最终解析值完整落入实验产物。

配置分为：

- 数据目录、市场和快照；
- 回测起止日、基准和初始资金；
- 策略及参数；
- 组合与风控约束；
- 撮合、滑点、容量和费用规则；
- 报告与日志选项。

密钥不属于普通配置。AKShare 无 token；可选 Tushare token 从环境变量或本机安全存储读取，日志中只显示“已配置/未配置”。

### 9.2 可观测性

结构化日志统一包含：`run_id`、模块、阶段、交易日、symbol（适用时）、耗时和错误类型。P0 记录数据请求、分区发布、每日回测进度和最终摘要。

错误分为：配置错误、权限/网络错误、数据质量错误、策略错误、账户不变量错误和基础设施错误。数据质量或账户不变量错误默认快速失败，不能吞掉后继续生成看似正常的净值。

### 9.3 恢复与幂等

- 数据同步以接口和分区为最小重试单元；
- 已验证且内容一致的分区不重复写；
- 临时文件完成并校验后再原子发布；
- 快照只有在所有必需分区通过质量门后才标记完成；
- 回测 P0 可以从头确定性重跑，P1 再增加按交易日检查点；
- 失败运行保留 manifest、日志和错误，不覆盖同名成功运行。

### 9.4 Conda 环境与跨机器部署

Conda 是唯一受支持的环境管理入口，不同时维护 Poetry、uv 或独立 `requirements.txt` 解析链。环境名固定为 `tbcaptial`。

`environment.yml` 是人工维护的唯一依赖声明，只放 Python 版本、直接依赖、channel 和必要的 pip 依赖。按当前决定不生成多平台 lock，也不做四平台预试装。声明保持平台无关；新机器真正使用时执行同一创建脚本和 smoke test，通过后把实际平台加入验收记录。

依赖规则：

- 统一使用 `conda-forge` 并启用严格 channel priority，避免混合 channel 引入难以解释的二进制组合；
- Python、NumPy、PyArrow、DuckDB 等原生/科学计算依赖优先使用 Conda 包；
- 只有 Conda 无合适包时才使用 `environment.yml` 中的 pip 子段；AKShare/Tushare 属于这一类；
- 项目本身在开发环境以 editable、`--no-deps` 方式安装，在部署环境安装带 hash wheel；
- 禁止临时 `conda install`/`pip install` 后不更新源声明；
- 更新直接依赖必须在当前目标平台重新更新环境、运行 smoke test，并记录关键版本变化；
- AKQuant 是特殊的二进制 backend：先完成 BT-000，固定 version/commit/wheel SHA-256 后再加入环境；不能依赖 `/Users/wa/akquant` 绝对路径。

部署审计契约由代码/构建产物 hash、`environment.yml` hash、当次 `conda list --explicit` hash、AKQuant backend hash、目标平台、数据快照 id 和运行配置 hash 组成。没有 lock 时，不同日期重新求解可能得到不同间接版本；实际包清单使差异可见，但不把它夸大为跨时间逐包锁定。

每次正式运行记录 Conda 版本、环境前缀、平台、Python、环境声明 hash、实际包清单和关键 backend 身份。当前实测基线见 `ENVIRONMENT_ACCEPTANCE.md`。

## 10. 测试策略

### 10.1 测试层次

| 层次 | 重点 | 运行频率 |
| --- | --- | --- |
| 单元测试 | 值对象、舍入、整手、费用、T+1、撮合和指标 | 每次提交 |
| 属性/不变量测试 | 任意事件序列下账户不变量、幂等和确定性 | 每次提交或夜间 |
| 契约测试 | AKShare 主/Tushare 辅映射、AKQuant adapter 和 schema 演进 | 每次提交，使用固定夹具 |
| 集成测试 | Parquet、DuckDB、快照、回测模块组合 | 每次 PR |
| 黄金测试 | 人工计算的小市场场景逐字段对比 | 每次 PR |
| 端到端测试 | 从固定原始夹具到最终报告 | 每次 PR/发布 |
| 性能测试 | 全市场多年日频的时间和内存基线 | 每周或重大改动 |

普通测试不直接依赖实时 AKShare/Tushare；在线集成测试由手工或定时任务运行，避免网络和上游变更导致普通 PR 不稳定。

### 10.2 P0 黄金场景

人工夹具覆盖至少 10 个交易日和 3 只股票，主动构造：

- 正常买入和卖出；
- 当日买入后尝试卖出；
- 买入不足一手；
- 卖出剩余零股；
- 停牌无行情；
- 涨停无法买入和跌停无法卖出；
- 现金不足导致缩量或拒绝；
- 最低佣金和卖出印花税；
- 一个复权因子变化日；
- 一个退市或股票池移除事件；
- 多订单争用现金的确定性排序。

黄金结果经独立人工复算确认后再固化。

## 11. 里程碑与优先级

以下工期是双人全职、需求不扩张时的工程估算，不是承诺日期。

### M0：设计冻结与工程基线（P0，1–2 天）

任务：

- 评审本计划，确认时间语义、资产范围和第一条策略；
- 确认 AKShare 主接口可用，并列出 Tushare 辅助接口权限；
- 完成 Conda 源声明、创建/激活脚本和当前开发机 smoke test；
- 固定 AKQuant `v0.3.2` backend 制品方案及兼容缺口；
- 确定本地数据根目录、磁盘容量、目录权限和可选的第二块本地磁盘备份位置；
- 将关键决策拆成 ADR；
- 定义黄金场景和完成标准。

验收：

- 无未解释的 P0 范围冲突；
- 两个人能用同一段话描述信号、订单和成交的时间关系；
- AKShare 主接口及 Tushare 辅助权限清单完成，不把字段缺口留到开发末期。
- 当前开发平台从 `environment.yml` 创建成功；按决定不做其他平台预试装。
- 本地数据目录权限正确，只有采集进程写数据分区，研究和回测只读已发布快照。

### M1：可信数据切片（P0，3–5 个开发日）

任务：

- 先实现配置、日志和 AKShare adapter，再实现 Tushare 辅助 adapter；
- 实现本地文件存储后端、单写者原子发布协议和 DuckDB catalog；
- 落地 Raw/Silver、主键、单位转换、schema 和 Parquet 分区；
- 同步股票、日历、日线、复权因子、涨跌停、停复牌和基准；
- 完成分区、幂等、质量门、manifest、快照和本地 DuckDB catalog 重建；
- 选定 3–20 只股票、约 3–6 个月的可核验切片。

验收：

- 同一批次重复同步不产生重复记录；
- AKShare/Tushare 原始响应分别能追溯到 Silver 行；
- 单位转换、停牌缺失和交易日完整性有自动测试；
- 快照冻结后，即使“最新数据”变化也可重读旧内容。
- 快照包导入新的本地数据根目录后，不调用 AKShare/Tushare 即可校验并查询同一数据；
- 删除本地 DuckDB 后，可从 snapshot manifest 完整重建且查询内容 hash 一致；
- 模拟采集中断、重复执行和上游修订时，不出现半发布快照或旧实验数据被覆盖。

### M2：AKQuant backend 适配与最小可信回测（P0，5–7 个开发日）

任务：

- 固定 AKQuant wheel/commit/hash，完成 backend 兼容 smoke；
- 实现 TBCaptial 公共端口、snapshot feed、Strategy bridge 和完整 `BarSlice`；
- 映射目标组合、T+1、整手、费用和账户结果；
- 补齐停牌/涨跌停终态拒单、跨生效日费用和公司行为门禁；
- 输出每日净值、订单、成交、持仓和拒单；
- 完成单元测试、不变量测试和黄金样例。

验收：

- 人工黄金场景逐字段一致；
- 任一交易日的现金和持仓都能从流水重放；
- 相同输入重复运行产物一致；
- 引擎无法访问决策时点之后的数据。

### M3：第一条策略迭代闭环（P0，3–5 个开发日）

任务：

- 完成特征、股票池、策略和组合接口；
- 实现横截面动量基线和一个单标的诊断策略；
- 完成绩效指标、实验 manifest 和静态报告；
- 提供一条命令完成快照选择、回测和报告；
- 建立基线结果，后续改动自动比较。

验收：

- 新策略只需实现策略接口和配置，不修改回测内核；
- 报告能解释成交失败、费用和仓位变化；
- 换机器或新环境后能基于同一快照复现实验；
- 从改策略到看到报告的人工操作不超过一条主命令。

### M4：全市场与研究效率（P1，2–4 周）

任务：

- 扩展到多年全市场日频；
- 历史股票池、ST/上市/退市、指数成分和公司行为；
- 特征缓存、因子分析、参数批量运行和实验对比；
- 行业/风格暴露、换手与容量诊断；
- profiling 后优化主要瓶颈。

验收：

- 不使用今天的上市名单回测历史；
- 关键财务/状态数据遵守 point-in-time；
- 全市场目标区间在约定机器资源内完成；
- 优化前后黄金结果不变，或差异有明确迁移说明。

### M5：模拟盘准备（P2，另行评审）

进入条件：M4 连续稳定、策略通过样本外验证、数据和账户对账成熟，并已选定券商与数据授权方案。

模拟盘不应直接复用研究脚本拼接实现。需要统一时钟、订单状态机、幂等 client id、断线恢复、交易前风控、日终券商对账、告警和人工熔断。

## 13. 性能预算与扩展原则

P0 不以极限性能为目标，但从第一天记录数据量、墙钟时间、峰值内存和主要阶段耗时。

建议初始工程目标：

- 小型黄金回测在 CI 中数秒内完成；
- 20 只股票、5 年日频单策略在开发机上交互式完成；
- 全市场 10 年日频的目标预算在 M4 根据真实硬件和策略复杂度确定；
- 单次实验不复制无法控制的全量 DataFrame；
- 因子按依赖和数据快照缓存，回测流式读取交易日切片。

优化顺序：减少重复 IO/计算 → 改善数据布局和向量化 → 批量并行实验 → Polars/Numba → 仅对确定热点使用 C++ 扩展。任何优化都必须通过黄金结果和数值容差检查。

## 14. 主要风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AKShare 上游变化/限流 | 主数据链路字段漂移或不稳定 | 固定响应契约；先 Raw；明确错误；版本化修订 |
| Tushare 辅助权限不足 | 复权、历史状态或涨跌停缺口 | M0 权限探针；缺失时只允许黄金夹具，不伪造正式快照 |
| AKQuant backend 漂移 | 回测语义或二进制部署不可控 | 固定 version/commit/wheel hash；adapter 隔离；黄金兼容矩阵 |
| 本地磁盘损坏、误删或空间耗尽 | 数据丢失或研究中断 | 磁盘空间预检、只追加目录、第二块本地磁盘备份和恢复演练 |
| DuckDB 文件并发访问 | 锁冲突或损坏 | catalog 可重建；单写多读；不放在 NAS 或同步盘 |
| Parquet 小文件膨胀 | 查询和文件枚举逐渐变慢 | 日期分区、月度合并、新版本发布和文件统计监控 |
| 幸存者偏差 | 历史收益严重高估 | 保存退市/暂停状态；使用按时点股票池；报告披露 |
| 未来函数 | 策略结果失真 | 统一 `available_time/as_of`；T+1 默认执行；时间边界测试 |
| 复权和公司行为错误 | 收益、持仓和费用无法对账 | 原始价格与因子分存；成交用原价；公司行为走账户事件 |
| 日线撮合过度乐观 | 涨跌停和大单策略收益虚高 | 保守可成交规则、参与率、滑点和拒单诊断 |
| 数据静默修订 | 同实验无法复现 | 不可变 Raw、版本化分区、快照 hash |
| 过早架构复杂化 | 回测迟迟不能跑 | P0 禁止微服务/分布式/实盘；按里程碑准入 |
| 过早 C++ 优化 | 构建与调试成本上升 | 必须先有 profiling 和性能预算，且保持 Python 参考实现 |
| 只看收益指标 | 错误策略被误判为有效 | 强制交易诊断、分阶段验证、基线与样本外评审 |

## 16. P0 完成定义

只有同时满足以下条件，才能宣告“回测与策略迭代已经跑起来”：

- 一条命令能基于指定数据快照运行基线策略并生成报告；
- 无需修改引擎即可新增另一条策略；
- 数据、配置、代码、随机种子和结果全部可追溯；
- 策略和回测路径不访问 AKShare/Tushare，只通过 snapshot id 读取本地已校验文件；
- 快照包复制到另一台机器后，可以离线导入并得到相同数据 hash；
- 无未来数据访问，股票池没有未经披露的幸存者偏差；
- T+1、整手、停牌、涨跌停、现金和费用已进入成交路径；
- 订单、成交、持仓、现金和净值可以逐日对账；
- 黄金场景、单元、集成和端到端测试通过；
- 同一输入重复运行得到一致结果；
- 报告明确披露日线撮合与公司行为的近似；
- 当前目标平台能从 `environment.yml` 创建环境并通过 smoke test；实际包清单和 AKQuant backend hash 已记录。

## 17. 后续决策门

完成 P0 后，不自动进入实盘。依次评审以下门槛：

1. **数据门**：历史状态、公司行为和 point-in-time 数据是否可信；
2. **研究门**：策略是否通过基线、成本、容量、样本外和稳健性检查；
3. **仿真门**：模拟盘成交、账户和券商对账是否长期稳定；
4. **风险门**：最大仓位、损失、异常数据、断线和人工熔断是否可控；
5. **实盘门**：数据授权、券商接口、监控、值守和小资金方案是否批准。

任何一个门未通过，都应继续修正当前层，而不是用下一层复杂度掩盖问题。
